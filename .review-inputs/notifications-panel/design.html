Notifications Panel Design

Proposed flow for a core bundled notifications surface: server-originated notifications, durable storage, panel live updates, optional Android-gated TTS, and session-aware navigation.

Core bundled panel/module Sources: tool / HTTP / CLI Newest first All / Unread filter Card / Compact view Stay until cleared TTS flag + Android gate

Locked product decisions

  • Notifications are created by server-side surfaces only: tool, HTTP, and CLI.
  • The notifications panel is a viewer and management surface, not a client-originated notification bus.
  • Each notification has title, body, createdAt, read state, optional sessionId, optional tts, and internal source.
  • Notifications are sorted newest first and persist until explicitly cleared.
  • Unread/read is represented both by state and by visual styling; the panel supports All and Unread filtering.
  • The panel supports Card and Compact density modes.
  • Tapping an item in either view toggles its read/unread state.
  • Compact view should expose a separate expand affordance so expansion does not steal the main tap action.
  • Bulk actions such as Mark all read and Clear all should live in an overflow menu rather than always-visible header buttons.
  • Notifications with a sessionId should expose a dedicated open/focus action on the session label or title. If invoked, it should focus an existing chat for that session when present; otherwise it should open a new chat tab.
  • Session-linked notifications should display a resolved session title in place of the raw id when available, with raw id fallback only when no title can be resolved.
  • tts: true is only a request. Android may speak it only when the device-side notification/TTS gate is enabled.

Recommended implementation shape

  • Create a core notifications module with a shared service layer.
  • Use three thin ingress adapters for tool, HTTP, and CLI.
  • Persist notifications in a server-side store behind a small repository/service API.
  • Broadcast live changes to panel clients over websocket, but likely with a dedicated notifications event class rather than only panel-local panel_event messages.
  • Use panel HTTP/operation calls for user actions such as mark-read and clear.
  • Use client layout context to locate an already-open chat panel for sessionId navigation.
  • Use client session summary/title context to resolve a human-readable session title for display, similar to existing chat/tab title behavior.
The design below assumes no client-originated notification creation over websocket. Websocket is used only for server → client live updates, and notifications likely need their own maskable event class so Android/native clients can subscribe explicitly.

Suggested TypeScript interfaces

type NotificationSource = 'tool' | 'http' | 'cli';

interface NotificationRecord {
  id: string;
  title: string;
  body: string;
  createdAt: string;      // ISO timestamp
  readAt: string | null;  // null = unread
  source: NotificationSource;
  sessionId: string | null;
  tts: boolean;
}

interface CreateNotificationInput {
  title: string;
  body: string;
  sessionId?: string | null;
  tts?: boolean;
}

interface NotificationListResult {
  notifications: NotificationRecord[]; // newest first
}
Read state recommendation: persist readAt instead of only a boolean. The panel can still derive isRead = readAt !== null, but the server keeps richer state.

}

{
  type: 'notification_event',
  event: 'removed',
  id: 'notif_123'
}

// Optional bulk resync path after reconnect/load
{
  type: 'notification_event',
  event: 'snapshot',
  notifications: NotificationRecord[]
}

// Web panel can still translate these into local UI updates.
// If helpful, the server may also emit panel_event messages for open notifications panels,
// but Android/native subscription support should not depend on panel_event alone.
Recommendation: add notification_event as a distinct websocket message type, then extend masked subscription handling so Android/native clients can explicitly opt into it. This keeps notification delivery parallel to other selectively subscribed event classes.

Panel behavior notes

  • Initial load fetches notifications from the server, sorted newest-first.
  • Filter state is local UI state: All vs Unread.
  • Density state is local UI state: Card vs Compact.
  • Main item tap toggles read/unread in either density mode.
  • Compact mode should provide a dedicated secondary control to expand one item for more detail.
  • Session-linked notifications should render a dedicated session label/title affordance that opens/focuses the session without overloading the main item tap.
  • Prefer showing a resolved session title from client session context; fall back to raw session id only when no title is known.
  • Read/unread should be shown through styling (outline, tint, dot, etc.), not heavy labels.
  • Source should primarily be conveyed by iconography; the stored source remains internal metadata.
  • If tts is true, show a subtle replay affordance in the item rather than treating it as a source or state icon.
  • Session-linked notifications should offer a single strong secondary action: open/focus session.
  • Bulk actions like mark-all-read and clear-all should live in an overflow menu to reduce clutter.

Implementation notes worth preserving

  • The existing app already supports server → panel live updates through panel_event.
  • The panel host already exposes activatePanel() and openPanel().
  • Client panel code can inspect host context such as panel.layout and session.summaries to resolve session-aware navigation.
  • This feature does not need client-originated notification creation over websocket.
